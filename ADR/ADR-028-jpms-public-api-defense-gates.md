# ADR-028: Three-layer JPMS public-API defense

Date: 2026-05-04
Status: Accepted

## Context

The SDK targets Maven Central as a JPMS-named module
(`io.github.mgrtomaszzurawski.ksef`). After publish the `module-info.java`
exports list becomes a binding contract — every package on it is
forever part of the consumer-visible compile surface. ADR-005 and
ADR-012 establish that internal packages (`sdk.internal.*`) and
generated code (`client.*`, `xml.*`) must NOT appear in that list.

Two slip categories produced repeated review findings during 0.x → 1.0:

1. **Surface drift.** A new public type added under `sdk.internal.*`
   would be invisible to consumers but still appear in Javadoc
   (because the Maven `excludePackageNames` list was string-matching
   `client.*` and missing the bare `client` package). Conversely,
   packages added to `module-info.java` `exports` could go
   undocumented if the Javadoc excludes weren't updated symmetrically.
2. **Consumer-side compile breakage.** Internal-typed parameters on
   public methods (e.g. `KsefSession(SessionClient, ...)`) compiled
   fine inside the SDK module but failed for downstream consumers
   trying to compile against the named module — the import would not
   resolve.

A single test was insufficient. Each gate covers a different drift
class and only the three together close the surface.

## Decision

Three layered gates, all running on every CI build:

### Gate 1: `ksef-jpms-consumer` compile-time surface check

A separate Maven module (`ksef-jpms-consumer`) declares its own
`module-info.java` requiring `io.github.mgrtomaszzurawski.ksef` and
contains a single class `JpmsConsumerCompileFixture` that touches a
representative slice of every public domain client, builder, and
record. The module compiles only if every reference resolves on the
modular path — i.e. every used type is in an exported package.

```java
// ksef-jpms-consumer/src/main/java/module-info.java
module io.github.mgrtomaszzurawski.ksef.jpms.consumer {
    requires io.github.mgrtomaszzurawski.ksef;
}
```

```java
// JpmsConsumerCompileFixture.java
import io.github.mgrtomaszzurawski.ksef.sdk.KsefClient;
import io.github.mgrtomaszzurawski.ksef.sdk.config.KsefEnvironment;
// ... 100+ public-API types touched, none from sdk.internal.* or client.*
```

A new public type added to a non-exported package fails this gate
immediately. Catches "internal type leaking via public method
signature" — the most common drift class.

### Gate 2: `JpmsExportsCoverageTest`

Parses `module-info.java` directly and asserts:

- every exported package contains at least one public class (no empty
  exports — usually a copy-paste error after a rename),
- every public `sdk.*` class lives in an exported package (no orphan
  public types invisible to consumers).

Lives in `ksef-client/src/test/java/.../api/`. Catches "added a
public type to a package not in module-info.java" — the second-most-common
drift class.

### Gate 3: `JavadocPackageGateTest`

Reads `target/reports/apidocs/element-list` (the Javadoc plugin's
package manifest) and asserts:

- documented Javadoc surface == JPMS exports exactly,
- no `sdk.internal.*` / `client.*` / `xml.*` package made it into
  Javadoc.

Catches "Maven `excludePackageNames` drifted from `module-info.java`"
— including the historical issue where `client.*` matched
`client.api` but missed the bare `client` package. The element-list
file is generated by the Javadoc plugin so this gate runs only after
`mvn package` (or `verify`); the test uses
`Assumptions.assumeTrue(Files.exists(...))` to gracefully skip on bare
`mvn test` runs.

## Alternatives considered

1. **Gate 1 only** — `ksef-jpms-consumer` catches signature leaks
   alone. Rejected: a new public type added to a package not on
   `module-info.java` exports compiles fine inside the SDK module and
   `ksef-jpms-consumer` won't reference it (because nobody documented
   its existence to write the import) — silent drift.
2. **Gate 2 only** — module-info parser. Rejected: the parser
   doesn't catch `excludePackageNames` Javadoc drift; consumers see
   the type on the Maven Central Javadoc surface even when JPMS
   blocks the import.
3. **Gate 3 only** — element-list comparison. Rejected: depends on
   `mvn package` having run; doesn't fire on local `mvn test`. Not
   defense-in-depth.
4. **One bigger gate combining all three checks.** Rejected: the
   three failure modes have different remediation paths
   (move file vs. edit module-info vs. fix `excludePackageNames`)
   and isolated tests give clearer error messages.

## Consequences

- **Adding a new public type requires updating four files**: the
  type itself, `module-info.java` exports, `JpmsConsumerCompileFixture`
  if the type is in a new package, and (transitively) the package's
  `package-info.java` for Javadoc. The three gates ensure the changes
  travel together.
- **`ksef-jpms-consumer` module is part of the main build** but not
  published — `<maven.deploy.skip>true</maven.deploy.skip>`. The
  gate runs at CI time, not in consumers' classpaths.
- **Gate 3 `Assumptions.assumeTrue` makes it skippable on bare
  `mvn test`.** Acceptable: the release pipeline runs `mvn verify`
  which triggers Javadoc generation, at which point the gate
  becomes mandatory. Local-dev `mvn test` users are not blocked by
  unrelated Javadoc plugin invocations.
- **`@since 1.0.0` discipline** layered on top: every public type
  carries a `@since` tag (269 retroactively added). Not enforced by
  these gates but tracked separately.

## Where it lives

- `ksef-jpms-consumer/` — separate module, `module-info.java` +
  `JpmsConsumerCompileFixture.java`.
- `ksef-client/src/test/java/.../api/JpmsExportsCoverageTest.java`
- `ksef-client/src/test/java/.../api/JavadocPackageGateTest.java`
- `ksef-client/pom.xml` — `maven-javadoc-plugin` `excludePackageNames`
  configuration (the configuration the Gate 3 test polices against
  drift).
